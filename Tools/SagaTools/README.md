# tools — SagaEngine tool dispatcher

> **Thin dispatcher.** Maps short logical names to executable paths and
> launches them as child processes. Knows nothing about Forge's build
> graph, nothing about Prism's parser, nothing about the engine. Owns
> a single artefact: a flat JSON registry.

---

## Quick start

**One command. That is the whole user experience.**

| OS | Command |
|----|---------|
| Linux / macOS | `python3 Tools/SagaTools/setup.py --all` |
| Windows (CLI) | `py Tools\SagaTools\setup.py --all` |
| Windows (Explorer) | double-click `Tools\SagaTools\setup.cmd` |

That single command does, in order:

1. Builds the `tools` dispatcher binary (`cmake` under the hood — invisible).
2. Stages it into `Tools/SagaTools/bin/tools` (`tools.exe` on Windows)
   alongside `bin/config/tools.registry.json` with absolute paths to
   every tool the dispatcher knows about — **no environment variables
   to set, ever**.
3. Runs `tools install forge` — builds the Forge binary into
   `Tools/Forge/tool/bin/`.
4. Runs `tools install prism` — sets up the Prism Python pipeline
   launcher (and the C++ extractor too, if LLVM is available).
5. Prints the one line you need to add `bin/` to your `PATH` for the
   current shell.

After PATH is set, this is what daily use looks like:

```bash
tools list                 # see what is registered and resolvable
tools forge --help         # Forge's own help
tools forge build          # delegates to forge build
tools prism --version      # delegates to prism
tools where forge          # print the resolved binary path
```

If you only want the dispatcher (no cascade), drop `--all`:

```bash
python3 Tools/SagaTools/setup.py     # dispatcher only
tools install forge                  # later, on demand
tools install prism
```

The **only** prerequisites are `cmake` (≥ 3.22) and Python 3.8+ on
PATH. Everything else — compilers, registry layout, environment
variables, `.exe` suffixes — the script handles for you.

### Troubleshooting

| Symptom | Cause / fix |
|---------|-------------|
| Paths in `tools list` show `u00fc` instead of `ü`. | Your project lives on a path with non-ASCII characters and the registry was generated by an old dispatcher. **Re-run `setup.py --all`** — the new bootstrap writes UTF-8 directly and the new dispatcher decodes `\uXXXX` properly. |
| `cmake: command not found`. | Install CMake ≥ 3.22 and add it to PATH (`brew install cmake` / `sudo apt install cmake` / `https://cmake.org/download`). |
| `tools forge` says "not available" but the binary clearly exists. | Re-run `setup.py --all` to regenerate the staged registry with the current absolute paths. |
| Project lives under OneDrive (`C:\Users\…\OneDrive\…`) and weird path errors appear. | Move the project to a non-OneDrive path (`C:\dev\SagaEngine` is conventional). OneDrive has been observed to rewrite paths under load and break tooling. |

## What `tools` is for

Developers and CI systems should not have to remember where every tool
lives, what it is called this week, or which environment variable points
to it. The `tools` dispatcher is the single entry point:

```text
tools list                     # what's available right now
tools install <name>           # run the registered bootstrap script
tools where <name>             # print the resolved binary path
tools <name> [args ...]        # forward args to the underlying tool
```

The first non-flag word after `tools` is treated as a tool name. Every
remaining argument is forwarded **verbatim** to the resolved
executable. The dispatcher does not parse those arguments, does not
rewrite them, does not inject anything of its own.

---

## What `tools` is NOT

| Not this | Because |
|----------|---------|
| **An orchestrator.** | The dispatcher does not chain tools, does not understand pipelines, does not pipe step N to step N+1. |
| **A build system.** | If you want a build, run `tools forge build`. The build logic is in Forge. |
| **A code indexer.** | If you want a graph, run `tools prism …`. The indexer is in Prism. |
| **A god-tool.** | If `tools` ever grows a `tools build` subcommand that internally invokes Forge with hard-coded flags, that subcommand has been smuggled in from the wrong layer. Reject it. |

The single load-bearing property: **deleting `tools` does not break
any tool it can launch.** Every executable in the registry remains
directly invocable on its own.

## CLI reference

```text
tools [--registry <path>] <command-or-tool> [args ...]

BUILT-IN COMMANDS
    list                     list all registered tools and resolved paths
    which <tool>             print the resolved path of <tool>, or '<unavailable>'
    where <tool>             alias for `which`
    run   <tool> [args ...]  explicit form of `tools <tool> [args ...]`
    install <tool>           run the registered bootstrap installer for <tool>
    --help, -h               print help and exit 0
    --version                print the dispatcher's version

ANY OTHER FIRST WORD is treated as a tool name and dispatched.
```

### Registry-file discovery order

1. `--registry <path>` flag.
2. `$SAGATOOLS_REGISTRY`.
3. `<exe-dir>/config/tools.registry.json` (where `setup.py` puts it).
4. `./tools.registry.json`.

### Registry format

A two-section JSON document. Both sections are flat name → string maps;
both are optional per tool.

```jsonc
{
  "schema_version": "1.0",
  "tools": {
    "forge": "${FORGE_BIN}",
    "prism": "${PRISM_BIN}"
  },
  "installers": {
    "forge": "../../Forge/tool/build.py",
    "prism": "../../Prism/build.py"
  }
}
```

Resolution rules per value:

- `${VAR}` substrings expand against the process environment.
- A value containing `/` or `\` is treated as a path, resolved relative
  to the registry-file directory if not already absolute.
- A bare name (no path separator) is searched on `PATH`, with
  platform-appropriate executable suffixes (`.exe`, `.bat`, `.cmd`) on
  Windows.

That is the entire spec. There are no per-tool option blocks, no
version constraints, no `default_args`, no chaining definitions. Adding
those is how dispatchers turn into orchestrators.

## Advanced — building from source by hand

If you want to skip `setup.py` and call `cmake` directly (for example
to integrate the dispatcher into an external CI pipeline):

```bash
cmake -S Tools/SagaTools  -B Tools/SagaTools/build  -DCMAKE_BUILD_TYPE=Release
cmake --build Tools/SagaTools/build --target tools
```

The binary lands at `Tools/SagaTools/build/tools`. To make `tools list`
work without `--registry`, copy `Tools/SagaTools/config/tools.registry.json`
to `<exe-dir>/config/tools.registry.json` (which is exactly what
`setup.py` does for you).

For Forge and Prism, their own bootstrap scripts work the same way:

```bash
python3 Tools/Forge/tool/build.py    # builds forge into Tools/Forge/tool/bin/
python3 Tools/Prism/build.py         # builds prism into Tools/Prism/bin/
```

## Repo-split readiness

The contents of `Tools/SagaTools/` become the new repo root with zero
rewrites. There is nothing inside it that includes a SagaEngine header,
links a SagaEngine library, or assumes the engine repository layout.
The same is true for `Tools/Forge/tool/` and `Tools/Prism/extractor/` +
`Tools/Prism/pipeline/`. All three projects can be split independently.
